# ISO 14229 UDS 基础概念培训 — 30–45 分钟逐页 Storyboard v0.2

用途：作为后续 `ppt-master` 高质量 PPT v1.x 的逐页制作蓝本。  
目标：把大纲细化到每页“讲什么、放什么、怎么动、用什么图表/视觉元素”。  
建议风格：沿用已通过样张的 **深色科技诊断仪表盘 / 车载网络拓扑 / 信息图**。

---

## 0. 总体制作约束

### 0.1 页面规模与时长

- 推荐页数：28 页。
- 标准授课：35–40 分钟。
- 扩展授课：45 分钟，加入练习和讨论。
- 平均节奏：普通概念页 60–90 秒，案例页 2–3 分钟，练习页 4–5 分钟。

### 0.2 标准依据优先级

1. **MCP 知识库标准原文**：
   - `Automotive_ISO_Standards/iso-14229-uds/iso-14229-1-2020.pdf`
   - `iso-14229-2-2013.pdf`
   - `iso-14229-3-2012.pdf`
   - `iso-15765-docan/*`
   - `iso13400-doip/*`
2. 公开教程仅作为教学表达参考：CSS Electronics、CANFD.net、TakeMind。
3. 真实项目中 DID/DTC/NRC/刷写流程需服从 OEM 诊断规范，本课程只讲基础框架。

### 0.3 视觉系统

- 背景：深色渐变、细网格、低透明度拓扑线。
- 主色：青蓝 `#22D3EE` / 深蓝 `#07111F` / 紫 `#7C3AED` / 橙 `#F59E0B` / 绿 `#10B981` / 红 `#F43F5E`。
- 字体：
  - 中文：Microsoft YaHei；
  - 英文/数字：Arial / Calibri；
  - SID/报文：Consolas。
- 禁止：大量白底卡片堆叠、过密 SID 表、底部小字堆满、Noto/JetBrains 等可能 fallback 的字体。

### 0.4 动画原则

PPT-master 导出阶段可先以静态矢量为主；若需要动画，使用 PowerPoint 后期增强或简单分步页替代。

推荐动画类型：

- **Appear/Fade**：逐步显示概念、服务组、报文字段。
- **Wipe left-to-right**：请求 → 响应、流程链路。
- **Morph/页面分步复制**：标准地图、刷写流程逐步展开。
- **Pulse/Glow**：当前 SID、错误 NRC、关键节点高亮。
- **Build sequence**：每页最多 3–5 个动画步骤，避免花哨。

### Slide 02 — 为什么汽车需要“问诊语言”？

**时长**：90 秒  
**目标**：用类比建立学习动机。

**屏幕文案**：

- 标题：`汽车也需要“问诊语言”`
- 三个问题：
  1. `ECU 现在状态正常吗？`
  2. `有没有故障码？`
  3. `能否进入刷写/维护模式？`

- 医生问诊需要标准问题，车辆诊断也一样；
- ECU 越多，统一语言越重要；

**视觉元素**：

- 中间：`统一问答语言` 桥梁。

**动画建议**：

- 先出现“医生问诊”类比；

- 最后用一条发光桥连起来。

**图表/数据类型**：类比图 / 双域映射图。

**依据**：教学类比，不作为标准事实。

### Slide 03 — 诊断仪到底在问 ECU 什么？

**图表/数据类型**：映射矩阵 / 业务-服务对照表。

**依据**：课程组织；服务类别来自 ISO 14229-1 功能单元目录。

### Slide 04 — UDS 是什么？

**时长**：90 秒  
**目标**：给出准确定义。

**屏幕文案**：

- `UDS = Unified Diagnostic Services`
- `ISO 14229 定义的统一诊断服务`
- `本质：Tester(Client) 与 ECU(Server) 的标准化请求-响应语言`
- 关键边界：`定义“说什么”，不是底层“怎么传”`

**讲师要点**：

- UDS 是服务集合，不是某一种线束或 CAN 报文格式；
- 它面向诊断系统通用需求；
- 它可以通过不同下层通信技术承载。

**视觉元素**：

- 中央大卡：`UDS`
- 三个环绕词：`Service`、`Request/Response`、`Application Layer`
- 底部小注：`ISO 14229-1:2020 — Application layer`

**动画建议**：

- `UDS` 中央节点出现；
- 三个关键词环绕点亮；
- 最后出现边界说明。

**图表/数据类型**：概念节点图。

**依据**：ISO 14229-1:2020 标题页与 Introduction。

---

### Slide 05 — UDS 不等于 CAN

**时长**：90 秒  
**目标**：消除最常见误解。

**屏幕文案**：

- 左：`UDS：应用层诊断服务语言`
- 右：`CAN / CAN FD / DoIP / LIN：承载与传输相关技术`
- 结论：`UDS on CAN ≠ UDS 就是 CAN`

**讲师要点**：

- 类比：UDS 是“语言”，CAN/DoIP 是“道路/快递”；
- UDS 可以跑在不同承载上；
- 这也是为什么 ISO 14229 有 UDSonCAN、UDSonIP、UDSonLIN 等实现相关部分。

**视觉元素**：

- 三层类比图：语言层 / 快递层 / 道路层；
- 或 OSI 分层简图。

**动画建议**：

- 先显示 UDS；
- 再下沉显示承载层；
- 用红色 `≠` 标记误解。

**图表/数据类型**：分层图 / 误解纠正图。

**依据**：ISO 14229-1:2020 Introduction 中 OSI 分层说明。

### Slide 06 — 标准地图：UDS 在车载通信栈里的位置

**时长**：2 分钟  
**目标**：把 ISO 14229 与 ISO 15765/13400/11898 等标准放到一张图里。

**屏幕文案**：

- `Application Layer：ISO 14229-1 / UDSonCAN / UDSonIP / UDSonLIN ...`
- `Session Layer：ISO 14229-2`
- `Transport/Network：ISO 15765-2 DoCAN / ISO 13400-2 DoIP / ISO 17987-2 LIN`
- `Data Link / Physical：ISO 11898 / ISO 13400-3 / ISO 17987-3/4`

**讲师要点**：

- 这是本课程最重要的边界图；
- ISO 14229-1 主要定义应用层服务；
- ISO 15765-2 解决 CAN 上多帧传输；
- ISO 13400-2 解决 IP 上诊断通信；
- 真实工程中经常把这些名字混在一起，必须分层。

**视觉元素**：

- OSI 7 层阶梯图；
- 每层一个发光横条；
- 当前课程聚焦区域用青色框选；
- 不同承载路径用竖向“通道”表示：CAN / DoIP / LIN。

**动画建议**：

1. 先显示 OSI 7 层空框；
2. 从上到下填入 ISO 14229-1、14229-2、15765-2、13400-2、11898；
3. 高亮应用层；
4. 淡出非重点，只保留学习路径。

**图表/数据类型**：标准分层图 / Sankey-like 层级图。

**依据**：ISO 14229-1:2020 Introduction 与 Table 1。

### Slide 08 — Client/Server 模型：Tester 问，ECU 答

**时长**：90 秒  
**目标**：建立基本通信角色。

**屏幕文案**：

- `Client / Tester：诊断仪、测试工具、产线设备、上位机`
- `Server / ECU：响应诊断请求的控制器`
- `Request → Response`
- `物理寻址 / 功能寻址：先知道有这个概念即可`

**讲师要点**：

- UDS 是典型请求-响应模型；
- Tester 主动发起请求，ECU 按规则响应；
- 一辆车可能有多个 ECU，寻址决定“问谁”。

**视觉元素**：

- 左侧 Tester；右侧多个 ECU 节点；
- 两条请求路径：单 ECU 物理寻址、多 ECU 广播式功能寻址。

**动画建议**：

- 先显示 Tester 和一个 ECU；
- 再展开多个 ECU；
- 请求线从 Tester 发出，目标 ECU 高亮。

**图表/数据类型**：网络拓扑图 / 请求响应链路图。

**依据**：ISO 14229-1:2020 Application layer services、service primitive 章节。

### Slide 09 — UDS 请求报文的基本结构

**时长**：90 秒  
**目标**：让学员能识别报文字段角色。

**屏幕文案**：

```text
Request = SID + SubFunction / Identifier / Parameters + Data
```

示例：

```text
10 03       进入扩展诊断会话
22 F1 90    读取一个 DID
19 xx ...   读取 DTC 信息
```

**讲师要点**：

- 不要一开始陷入所有字节含义；
- 先识别第一个字节通常是服务标识 SID；
- 后面的字段取决于服务类型。

**视觉元素**：

- 报文字节拆解条；
- 不同字段用颜色块：SID 青色、子功能紫色、参数橙色、数据绿色。

**动画建议**：

- 报文整体出现；
- 字段分段高亮；
- 示例报文逐条替换。

**图表/数据类型**：字节结构分解图。

**依据**：ISO 14229-1:2020 Application layer protocol、SI/service identifier、request message 章节。

### Slide 10 — SID 是什么？

**时长**：75 秒  
**目标**：理解 SID 是服务入口。

**屏幕文案**：

- `SID = Service Identifier`
- `它告诉 ECU：我要调用哪一类诊断服务`
- 代表例子：
  - `0x10` 会话控制
  - `0x22` 读数据
  - `0x19` 读故障码
  - `0x34/0x36/0x37` 下载与传输

**讲师要点**：

- 十六进制只是表示方式；
- 初学阶段只记代表性 SID；
- 不要把课程变成 SID 背诵。

**视觉元素**：

- “SID 芯片墙”：每个 SID 是一个发光 chip；
- 中间大字：`SID → Service`。

**动画建议**：

- SID chip 逐个点亮；
- 点亮时出现中文服务名。

**图表/数据类型**：标签墙 / Chip map。

**依据**：ISO 14229-1:2020 SI/service identifier 章节；服务目录。

### Slide 11 — 子功能与参数：同一个服务下的不同问法

**时长**：90 秒  
**目标**：理解服务不是只有一个固定动作。

**屏幕文案**：

- `Service = 大类`
- `SubFunction / Parameter = 具体问法`
- 示例：
  - `0x10`：默认 / 扩展 / 编程会话
  - `0x19`：按状态读 DTC / 读快照 / 读扩展数据
  - `0x31`：启动 / 停止 / 查询例程结果

**讲师要点**：

- 相同 SID 下，不同子功能代表不同动作；
- 参数决定读哪个 DID、哪个 DTC、哪个例程；
- 测试用例必须写清参数，不只写服务号。

**视觉元素**：

- 树状结构：SID 根节点 → 子功能 → 参数；
- 三个示例树并排。

**动画建议**：

- 先显示 SID 根节点；
- 展开子功能枝干；
- 最后显示参数叶子。

**图表/数据类型**：树图 / Drill-down 图。

**依据**：ISO 14229-1:2020 request message SubFunction parameter、各服务章节。

### Slide 12 — 正响应：为什么常见是 SID + 0x40？

**时长**：90 秒  
**目标**：掌握正响应识别。

**屏幕文案**：

```text
请求：10 03        响应：50 03
请求：22 F1 90     响应：62 F1 90 <data>
```

提示：`很多服务的正响应 SID = 请求 SID + 0x40`

**讲师要点**：

- 正响应不只是“OK”，还会带回关键数据；
- 响应通常会回显关键标识；
- 这是报文识别中的快速判断方法，但具体服务仍要查规范。

**视觉元素**：

- 左侧请求字节，右侧响应字节；
- `+0x40` 箭头；
- 响应数据尾部用绿色 payload 区。

**动画建议**：

- 请求出现；
- `+0x40` 运算动画；
- 响应字节亮起。

**图表/数据类型**：报文字节转换图。

**依据**：ISO 14229-1:2020 positive response message 章节。

### Slide 13 — 负响应与 NRC：ECU 拒绝也有规则

**时长**：2 分钟  
**目标**：建立 NRC 意识。

**屏幕文案**：

```text
Negative Response = 7F + 原请求SID + NRC
```

常见拒绝原因：

- 服务不支持；
- 会话不对；
- 安全未解锁；
- 条件不满足；
- 参数格式错误。

**讲师要点**：

- 负响应不是“没通信上”；
- 它是 ECU 按规则告诉你“为什么不能执行”；
- 测试时必须验证负响应分支，否则就是表面覆盖。

**视觉元素**：

- 红色/橙色错误响应面板；
- `7F 27 35` 示例；
- NRC 原因列表像故障诊断告警板。

**动画建议**：

- 正常请求线被 ECU 拒绝；
- 返回红色 `7F` 包；
- NRC 原因逐条出现。

**图表/数据类型**：错误码解释卡 / 状态告警面板。

**依据**：ISO 14229-1:2020 Negative response service identifier、NRC 章节。

### Slide 14 — 服务地图：不要从背表开始

- 初学阶段不追求完整记忆。

**视觉元素**：

- 六边形服务地图；
- 中心 `UDS Service Map`；
- 周围六个功能域。

**动画建议**：

- 中心出现；
- 六个功能域环形展开。

**图表/数据类型**：功能地图 / Hub-and-spoke。

**依据**：ISO 14229-1:2020 目录第 10–15 章功能单元。

### Slide 15 — 服务组 1：会话与连接保持

**时长**：90 秒  
**目标**：理解会话是许多服务的前置上下文。

**屏幕文案**：

- `0x10 DiagnosticSessionControl`
- `0x3E TesterPresent`
- `0x11 ECUReset`
- 典型场景：`默认会话 → 扩展会话 → 编程会话`

**讲师要点**：

- ECU 在不同会话下开放不同能力；
- 长时间诊断需要保活；
- 某些操作后可能需要 ECU reset。

**视觉元素**：

- 会话状态机：Default / Extended / Programming；
- 保活心跳线；
- Reset 图标。

**动画建议**：

- 会话节点按顺序切换；
- TesterPresent 作为心跳闪烁；
- Reset 节点最后短暂高亮。

**图表/数据类型**：状态机 / 心跳图。

**依据**：ISO 14229-1:2020 第 10 章 Diagnostic and communication management functional unit。

### Slide 16 — 服务组 2：权限与安全

**时长**：90 秒  
**目标**：理解不是所有操作都能直接执行。

**屏幕文案**：

- `0x27 SecurityAccess`
- `0x29 Authentication（2020 版新增服务）`
- 概念：`Challenge / Response · Seed / Key · 权限门禁`

**讲师要点**：

- 刷写、写配置、执行关键例程通常需要权限；
- SecurityAccess 可类比门禁挑战-应答；
- ISO 14229-1:2020 新增 Authentication，反映网络安全主题增强；
- 本课不展开算法和证书细节。

**视觉元素**：

- 锁门图；
- Tester 请求 seed，ECU 返回 challenge；
- Tester 返回 key，门打开。

**动画建议**：

- 锁住状态；
- seed/key 两次报文往返；
- 解锁后流程线变绿。

**图表/数据类型**：挑战-应答序列图。

**依据**：ISO 14229-1:2020 Foreword 新增 Authentication；第 10.4 SecurityAccess、第 10.6 Authentication。

### Slide 17 — 服务组 3：读写数据 DID

**时长**：90 秒  
**目标**：理解 DID 是“数据项地址/编号”。

**屏幕文案**：

- `0x22 ReadDataByIdentifier`
- `0x2E WriteDataByIdentifier`
- 示例：VIN、软件版本、配置、传感器/状态信息
- 关键概念：`DID = Data Identifier`

**讲师要点**：

- DID 像数据字典中的编号；
- 0x22 是读取，0x2E 是写入；
- 具体 DID 由标准/OEM/项目定义，不能随便猜。

**视觉元素**：

- 数据字典表：DID、名称、返回数据；
- 报文示例 `22 F1 90` → `62 F1 90 ...`。

**动画建议**：

- 先显示 DID 字典；
- 高亮 `F1 90`；
- 展开返回 payload。

**图表/数据类型**：数据字典表 / 报文解码卡。

**依据**：ISO 14229-1:2020 第 11 章 Data transmission functional unit。

---

### Slide 18 — 服务组 4：故障诊断 DTC

**时长**：2 分钟  
**目标**：理解读 DTC 与清 DTC，以及状态信息。

**屏幕文案**：

- `0x19 ReadDTCInformation`
- `0x14 ClearDiagnosticInformation`
- DTC 信息通常包括：`DTC 编码 · 状态位 · 快照 · 扩展数据`
- 提醒：`读 DTC ≠ 清 DTC`

**讲师要点**：

- DTC 是 ECU 记录的故障信息；
- 读取 DTC 可以有多个子功能；
- 测试时不能只看“有响应”，还要看状态和数据是否正确；
- 清除 DTC 是破坏性动作，前置条件要明确。

**视觉元素**：

- 故障码仪表盘；
- DTC 列表表格：DTC、状态、快照、扩展数据；
- 读/清两个按钮，清除按钮带警示色。

**动画建议**：

- DTC 告警灯出现；
- 读取动作展开 DTC 表；
- 清除动作需要确认门槛。

**图表/数据类型**：故障码表 / 状态位矩阵。

**依据**：ISO 14229-1:2020 第 12 章 Stored data transmission functional unit；本地 0x19 微课样例。

---

### Slide 19 — 服务组 5：IO 控制与例程控制

**时长**：90 秒  
**目标**：理解 UDS 不只是读数据，也能触发动作。

**屏幕文案**：

- `0x2F InputOutputControlByIdentifier`
- `0x31 RoutineControl`
- 典型动作：执行器控制、自检、擦除、校验、学习例程
- 风险提示：`动作类服务更依赖会话、权限和条件`

**讲师要点**：

- 0x2F 控制输入输出；
- 0x31 启动/停止/查询例程；
- 刷写流程里的擦除/校验常用例程控制表达。

**视觉元素**：

- 左：执行器开关面板；
- 右：例程任务队列：Start / Stop / Result。

**动画建议**：

- 开关从 Off 到 On；
- Routine 进度条运行；
- Result 返回绿/红状态。

**图表/数据类型**：控制面板 / 任务状态图。

**依据**：ISO 14229-1:2020 第 13、14 章。

---

### Slide 20 — 服务组 6：刷写与数据传输

**时长**：90 秒  
**目标**：理解上传/下载服务族。

**屏幕文案**：

- `0x34 RequestDownload`
- `0x36 TransferData`
- `0x37 RequestTransferExit`
- `0x38 RequestFileTransfer`
- 结论：`刷写是一条受控传输流程，不是直接发文件。`

**讲师要点**：

- RequestDownload 协商下载；
- TransferData 分块传输；
- RequestTransferExit 结束传输；
- 文件传输/刷写还要结合校验、例程、重启。

**视觉元素**：

- 文件被切成数据块；
- 数据块沿链路传到 ECU flash 区；
- 最后一块触发 transfer exit。

**动画建议**：

- 文件分块；
- 数据块依次移动；
- 传输完成后 ECU 节点变绿。

**图表/数据类型**：分块传输流程图。

**依据**：ISO 14229-1:2020 第 15 章 Upload download functional unit。

### Slide 21 — 案例 1：读取 VIN / 软件版本

**时长**：2 分钟  
**目标**：把 0x22 和 DID 读数据讲成完整案例。

**屏幕文案**：

```text
Tester → ECU: 22 F1 90
ECU → Tester: 62 F1 90 <VIN data>
```

三个识别点：

1. `22`：读 DID；
2. `F1 90`：目标 DID；
3. `62`：正响应。

**讲师要点**：

- 这类服务常用于读取车辆信息、软件版本、配置状态；
- 返回数据不是直接“看懂”，需要 DID 定义；
- 测试用例要写清目标 DID、期望长度和解码规则。

**视觉元素**：

- 报文解码卡；
- VIN 数据条被拆成 ASCII 字符；
- DID 字典浮窗。

**动画建议**：

- 请求报文逐字段拆解；
- 响应 payload 展开为 VIN 字符串；
- DID 字典标出来源。

**图表/数据类型**：报文解码图 / 字段映射表。

**依据**：ISO 14229-1:2020 ReadDataByIdentifier；公开教程 VIN 示例仅作表达参考。

### Slide 22 — 案例 2：读取 DTC

**时长**：2–3 分钟  
**目标**：把 DTC 读法与测试关注点连接起来。

**屏幕文案**：

- `0x19 ReadDTCInformation`
- 测试关注：
  - 前置故障是否注入？
  - DTC 编码是否正确？
  - 状态位是否正确？
  - 快照/扩展数据是否符合预期？

**讲师要点**：

- 读 DTC 是诊断测试中非常常见的场景；
- 只收到正响应不代表测试充分；
- 要看 DTC 状态变化和证据链。

**视觉元素**：

- DTC 生命周期小图：故障注入 → 检测 → 存储 → 读取 → 清除；
- DTC 结果表：DTC、StatusMask、Snapshot、ExtData。

**动画建议**：

- 生命周期逐步点亮；
- 读取后表格出现；
- 可审计检查点逐条打勾。

**图表/数据类型**：生命周期流程图 / DTC 结果表。

**依据**：ISO 14229-1:2020 ReadDTCInformation；本地 0x19 微课经验。

### Slide 23 — 案例 3：ECU 刷写流程

**时长**：3 分钟  
**目标**：串联多类服务。

**屏幕文案**：

```text
0x10 编程会话
→ 0x27 安全解锁
→ 0x31 擦除/校验
→ 0x34 请求下载
→ 0x36 数据传输
→ 0x37 退出传输
→ 0x31 完整性校验
→ 0x11 ECU 重启
```

**讲师要点**：

- 刷写不是一个服务完成；
- 它组合了会话、权限、例程、下载、传输、校验、重启；
- 任一步条件不满足都可能负响应；
- 这是理解 UDS 服务协作的最好案例。

**视觉元素**：

- 横向发光流程链；
- 每个节点一个 SID chip；
- 下方分组色带：会话/权限/例程/传输/重启。

**动画建议**：

- 流程节点按顺序亮起；
- 数据包在 0x36 阶段连续流动；
- 校验成功后 ECU 重启脉冲。

**图表/数据类型**：流程链 / 泳道式服务组合图。

**依据**：ISO 14229-1:2020 Upload download functional unit、RoutineControl、Non-volatile server memory programming process。

### Slide 24 — 小练习：看三条报文，猜它在干什么

**时长**：4–5 分钟  
**目标**：检验理解。

**屏幕文案**：

```text
A. 10 03  -> 50 03
B. 22 F1 90 -> 62 F1 90 ...
C. 7F 27 35
```

问题：

1. 请求还是响应？
2. 哪个服务？
3. 成功还是拒绝？
4. 大概业务场景？

**讲师要点**：

- A：会话控制正响应；
- B：读 DID 正响应；
- C：安全访问负响应，具体 NRC 需查规范/项目定义；
- 重点不是背答案，而是建立读报文方法。

**视觉元素**：

- 三张报文挑战卡；
- 右侧答案面板先隐藏；
- 底部计时/互动提示。

**动画建议**：

- 先显示题目；
- 暂停讲解；
- 点击后逐张翻卡显示答案。

**图表/数据类型**：互动练习卡 / 翻牌式答案。

**依据**：前面报文结构与正/负响应概念。

### Slide 26 — 从“表面覆盖”到“可审计覆盖”

**时长**：2–3 分钟  
**目标**：体现测试/ASPICE 审计价值。

**屏幕文案**：

低质量写法：

```text
发送 0x22，检查收到正响应。
```

可审计写法：

```text
前置会话/权限明确
请求 DID 明确
响应 SID/长度/字段/解码明确
负响应分支明确
证据可回放
```

**讲师要点**：

- 用户特别反感“表面覆盖”，这里要明确讲；
- UDS 测试要覆盖前置条件、请求参数、响应字段、负响应、时序/状态；
- DTC 和刷写场景尤其需要证据链。

**视觉元素**：

- 左右对比：表面覆盖 vs 可审计覆盖；
- 右侧有 checklist：前置、请求、响应、负响应、证据。

**动画建议**：

- 左侧低质量示例先出现并打叉；
- 右侧逐条打勾；
- 最后出现“可审计证据链”。

**图表/数据类型**：Before/After 对比表 / QA checklist。

**依据**：测试方法论与用户偏好；可连接 ASPICE SYS.5/测试审计。
